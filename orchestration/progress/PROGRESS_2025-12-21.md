## Summary

Implemented OpenMP parallelization of tensor repacking in llama.cpp, achieving **2.2x faster model loading** on 96-core EPYC. Submitted upstream PR and updated local benchmark infrastructure.

---

## 1. Parallel Tensor Repacking Patch

### Problem
Model loading was bottlenecked by single-threaded tensor repacking for AVX-512 optimization. For a 271GB model, ~120s of 150s load time was spent in `repack_*` functions.

### Solution
Added `#pragma omp parallel for` to 6 repack functions in `ggml/src/ggml-cpu/repack.cpp`:
- `repack_q4_0_to_q4_0_4_bl`
- `repack_q4_K_to_q4_K_8_bl` (most common - Q4_K_M models)
- `repack_q2_K_to_q2_K_8_bl`
- `repack_q4_0_to_q4_0_8_bl`
- `repack_iq4_nl_to_iq4_nl_4_bl`
- `repack_iq4_nl_to_iq4_nl_8_bl`

### Benchmark Results

| Model Size | Before | After | Speedup |
|------------|--------|-------|---------|
| 6.8GB Q4_K | 5.0s | 3.3s | **1.5x** |
| 19GB Q4_K | 11.9s | 5.3s | **2.2x** |
| 271GB Q4_K | ~150s | ~60s | **~2.5x** (estimated) |

## 2. Benchmark Infrastructure Fix

### Problem
Benchmark scripts set `OMP_NUM_THREADS=1` which disabled parallel repack AND hurt prompt processing speed.

### Solution
Removed `OMP_NUM_THREADS=1` from `scripts/lib/executor.py`.

### Impact
- Parallel repack now works automatically
- Prompt processing improved 2.4x (49 → 119 t/s)
- Generation speed unchanged (~19.8 t/s)

## 3. Server Mode (from earlier session)

Added `--server-mode` flag to `run_benchmark.py`:
- Keeps model resident in RAM via llama-server
- Sends HTTP requests instead of spawning processes
- Eliminates model reload between questions

Not enabled by default - use `--server-mode` flag when desired.
